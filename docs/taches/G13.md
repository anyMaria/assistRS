## Le bug à corriger (comprendre avant de coder)

Aujourd'hui, l'ingestion des résultats Apify se fait **uniquement** dans
`GET /api/inspiration/statut`, appelée par le polling du navigateur (`PollStatus`,
toutes les 5 s). Conséquence : si Ana ferme l'onglet pendant le run (~100 s), personne
n'appelle plus la route, et la recherche reste `en_cours` **pour toujours**. C'est la
cause principale du « ça ne marche pas tout le temps ».

## 1. Extraire l'ingestion dans un module partagé

- Créer `src/lib/inspiration-ingest.ts` avec une fonction
  `ingestSearch(searchId: number): Promise<{ status: string; errorMessage: string | null }>`
  qui reprend **exactement** la logique actuelle de la route statut : lire la recherche,
  interroger le statut du run Apify, si `SUCCEEDED` → récupérer le dataset, normaliser,
  insérer les `inspiration_items`, logger le coût, passer en `termine` ; si échec →
  `erreur`. La route `statut` est réécrite pour appeler cette fonction (elle ne garde
  que la lecture des query params et la réponse JSON).
- **Idempotence obligatoire** : la fonction ne fait rien si la recherche n'est plus
  `en_cours` (elle renvoie juste l'état actuel). C'est ce qui protège contre les appels
  concurrents webhook + polling + retries Apify.

## 2. Webhook Apify — ingestion côté serveur, page fermée ou pas

- Nouvelle route `POST /api/inspiration/webhook`. Sécurité : query param
  `?secret=<CRON_SECRET>` (réutilise la variable existante, pas de nouvelle clé) ;
  toute requête sans le bon secret → 401 sans détail.
- Dans `startApifyRun` (`src/lib/apify.ts`) : au démarrage du run, attacher un webhook
  *ad hoc* (param `webhooks` de l'API « run actor », tableau JSON encodé en base64)
  avec `eventTypes: ["ACTOR.RUN.SUCCEEDED", "ACTOR.RUN.FAILED", "ACTOR.RUN.TIMED_OUT",
  "ACTOR.RUN.ABORTED"]` et `requestUrl` =
  `${APP_URL}/api/inspiration/webhook?secret=${CRON_SECRET}`.
  - `APP_URL` : nouvelle variable d'env (URL publique de l'app, ex.
    `https://assistrs.vercel.app`). Si absente, retomber sur
    `https://${process.env.VERCEL_PROJECT_PRODUCTION_URL}` ; si rien de tout ça
    (dev local), **ne pas attacher de webhook** — le polling existant couvre le dev.
  - L'échec d'attachement du webhook ne doit **jamais** faire échouer le démarrage du
    run (try/catch, log console).
- Le handler webhook : retrouver la recherche par `apifyRunId` (payload du webhook —
  vérifie le nom exact du champ dans la doc, normalement `resource.id`), puis appeler
  `ingestSearch`. Réponse 200 même si la recherche est déjà traitée (idempotence §1) —
  Apify retente sinon.
- **Garder le polling client** (`PollStatus`) tel quel : c'est le fallback en dev et
  ça rafraîchit l'UI quand la page est ouverte. Les deux chemins passent par la même
  fonction d'ingestion, donc pas de double insertion possible.

## 3. Rattrapage des recherches zombies + retry automatique

- Migration Drizzle : colonne `retry_count` (integer, défaut 0) sur
  `inspiration_searches`. `npm run db:push` (piège §11.6).
- **Rattrapage** : fonction `rattraperRecherchesBloquees()` dans
  `src/lib/inspiration-ingest.ts` — toute recherche `en_cours` avec `apifyRunId`
  créée il y a **plus de 10 minutes** → appeler `ingestSearch` (qui interroge Apify
  et conclut : terminé, erreur, ou toujours en cours). L'appeler : 1) dans le cron
  quotidien ; 2) au chargement de l'onglet S'inspirer (`InspirerTab`), en tâche de
  fond non bloquante. Ça répare aussi les recherches déjà bloquées en base.
- **Retry** : quand un run se conclut en échec (`FAILED`/`TIMED-OUT`/`ABORTED`) et que
  `retry_count === 0` → relancer automatiquement **un** nouveau run Apify (mêmes
  paramètres), mettre à jour `apifyRunId`/`apifyDatasetId`, incrémenter `retry_count`,
  rester `en_cours`. Au deuxième échec → statut `erreur`, message existant. Ne jamais
  logger de coût pour un run échoué.

## 4. Pré-chauffe hebdomadaire par le cron (la lenteur devient invisible)

- Migration Drizzle : colonne `origin` (text, défaut `"manuelle"`) sur
  `inspiration_searches` — valeurs `manuelle | cron`.
- Dans `/api/cron/quotidien` (piège §11.8 : un seul cron), **seulement le lundi**
  (Europe/Paris) : pour chaque marque avec une ligne éditoriale, prendre ses piliers
  (`brand_editorial.pillars`, JSON `[{name}]`) et lancer une recherche **Pinterest
  uniquement** (source vérifiée, la moins chère) par pilier, avec `origin: "cron"`.
  Garde-fous, dans l'ordre :
  1. cache < 14 j sur `normalizedTheme` + source → ne rien lancer (comportement
     existant) ;
  2. plafond global : **6 recherches cron maximum par semaine**, toutes marques
     confondues (les premières marques/piliers par ordre de création) ;
  3. budget : si dépense Apify du mois ≥ `BUDGET_SOFT_LIMIT_CENTS`, **ne rien lancer**
     et logger en console (pas de confirmation possible dans un cron).
  Les résultats arrivent par le webhook §2 — le cron ne poll pas.
- UI : en tête de l'onglet S'inspirer, section « Suggestions de la semaine » : les
  recherches `origin = "cron"` terminées des 14 derniers jours, affichées en chips
  cliquables `{pilier} · {marque} · {n} visuels` qui ouvrent les résultats (même
  mécanisme que le param `?recherche=` existant). Rien à afficher → pas de section.

## 5. Pertinence — Gemini en amont (requête) et en aval (tri)

Rappel : SDK `@google/genai` (piège §11.11), coût 0 ¢ (palier gratuit) mais logger
quand même dans `api_usage` (`service: "gemini"`). Chaque appel Gemini est optionnel :
**tout échec Gemini doit dégrader vers le comportement actuel**, jamais bloquer.

### 5a. Requête optimisée par source (remplace le tronquage à 3 mots)

- Créer `src/lib/inspiration-query.ts` : `optimizeQuery(theme, source, account?)` →
  un appel Gemini avec `responseSchema` JSON (piège §11.10 : JSON structuré, pas de
  texte libre) qui produit la requête idéale pour la source, à partir du thème + du
  contexte de marque (secteur, positionnement si disponibles) :
  - `pinterest` : 2-3 mots-clés visuels, sans accents (l'anglais marche souvent mieux —
    laisser Gemini choisir) ;
  - `instagram` : **un seul hashtag** sans `#` ni espaces ;
  - `linkedin` / `facebook` / `metaAds` : 2-4 mots-clés simples.
  Prompt-système : préambule commun du GUIDELINE §5.1.
- Dans `startApifyRun` : utiliser `optimizeQuery` ; en cas d'échec → `normalizeQuery`
  actuel (fallback). **La clé de cache ne change pas** : `normalizedTheme` reste
  `normalizeQuery(theme)` — seule l'entrée envoyée à l'actor change.
- Migration : colonne `query_used` (text, nullable) sur `inspiration_searches` pour
  tracer la requête réellement envoyée ; l'afficher en petit dans l'en-tête de
  résultats (`recherché : « branding artisan »`).

### 5b. Tri de pertinence des résultats

- Migration : colonnes `title` (text, défaut `""`) et `relevant` (integer boolean,
  défaut `true`) sur `inspiration_items`.
- Dans `normalizeItem` (`src/lib/apify.ts`) : capturer aussi un `title` texte quand la
  source en fournit un (Pinterest `name`, LinkedIn/Facebook texte du post tronqué à
  200 caractères, Meta Ads texte de l'annonce). Lance un run de test par source pour
  vérifier les noms de champs réels (GUIDELINE §6).
- Après ingestion (dans `ingestSearch`, quand il y a des items) : **un seul** appel
  Gemini pour tout le lot — entrée : thème + liste `{id, title, author}` ; sortie
  JSON `[{id, score}]` (0-10, pertinence par rapport au thème). Items avec score < 4 →
  `relevant = false`. Échec Gemini → tout reste `relevant = true`.
- UI : les items `relevant = false` sont masqués par défaut ; s'il y en a, bouton
  discret « Voir aussi {n} résultats hors sujet » qui les déplie (état client local,
  pas de rechargement). Afficher le `title` sur la carte quand il existe.

- [ ] `npx tsc --noEmit` et `npm run build` passent.
