## Le principe (à respecter dans toutes les décisions)

Buffer **copie les médias chez lui** à la création du post : nos fichiers ne servent
plus à rien après l'envoi. On fait donc de l'**hébergement transitoire**, pas une
médiathèque : upload au moment de planifier → envoi à Buffer → purge automatique
7 jours après. Le stockage ne contient jamais plus que la semaine en cours.

Deux règles d'Ana, non négociables :
1. **Aucune conversion/compression automatique des visuels** (qualité = son métier).
   On envoie le fichier tel quel ; les problèmes de format donnent des
   *avertissements*, jamais des transformations.
2. **Pas de vidéo en v1** : les reels se finalisent dans Buffer directement
   (un reel = 50-200 Mo, le quota Blob gratuit y passerait en quelques posts).

## 1. Table `publication_assets`

Migration Drizzle (`npm run db:push`, piège §11.6) :

```
publication_assets
  id            integer PK autoincrement
  publicationId integer NOT NULL → publications.id (onDelete cascade)
  url           text NOT NULL      -- URL publique du blob (Buffer doit pouvoir la télécharger)
  pathname      text NOT NULL      -- pour la suppression via del()
  position      integer NOT NULL DEFAULT 0   -- ordre dans le carrousel
  sizeBytes     integer NOT NULL DEFAULT 0
  createdAt     timestamp DEFAULT unixepoch
```

Contrairement aux blobs privés existants (moodboards, rapports), ces blobs sont
**`access: "public"`** — Buffer télécharge l'URL, un proxy authentifié le bloquerait.
C'est acceptable : ces visuels sont destinés à être publiés publiquement, et
`addRandomSuffix: true` rend l'URL non devinable. Préfixe de chemin :
`publications/{publicationId}/…`.

## 2. Upload côté client dans le formulaire de publication

- Route `POST /api/publications/upload` avec `handleUpload` de `@vercel/blob/client` :
  - `onBeforeGenerateToken` : `allowedContentTypes: ["image/jpeg", "image/png",
    "image/webp", "image/gif"]`, `maximumSizeInBytes: 20 Mo`, `addRandomSuffix: true` ;
    passer `publicationId` dans `clientPayload`.
  - `onUploadCompleted` : insérer la ligne `publication_assets` (position = max + 1).
    ⚠️ En dev local, `onUploadCompleted` n'est pas rappelé par Vercel (URL non
    joignable) — prévoir le fallback documenté par Vercel (enregistrement via
    l'action serveur au retour de `upload()` côté client si la ligne n'existe pas).
- Composant client `AssetUploader` (`src/components/AssetUploader.tsx`), intégré à
  `PublicationForm` **à la place** du champ texte `visualUrl` (garder le champ URL en
  repli discret « ou coller une URL » pour compat et pour le cas Blob non configuré) :
  - input fichier multiple `accept="image/*"` + zone de dépôt ;
  - vignettes des assets existants, réordonnancement par flèches ◀▶ (met à jour
    `position`), suppression par icône (supprime la ligne **et** le blob via `del`) ;
  - compteur « {n}/10 » — maximum **10 images** (limite carrousel Instagram) ;
  - mention : « Reels et vidéos : finalise-les directement dans Buffer. »
- L'upload nécessite une publication existante (il faut un `publicationId`) : dans le
  formulaire de **création**, la section médias affiche « Enregistre d'abord la
  publication, puis ajoute les visuels » ; elle n'est active qu'en modification.
  (Ne complique pas avec des brouillons — flux réel d'Ana : planifier, puis produire
  le visuel plus tard, cf. checklist de production existante.)

## 3. `envoyerSurBuffer` : plusieurs assets, vrais contrôles

- Charger les `publication_assets` de la publication, triés par `position`.
  Construire `assets` pour `createPost` : chaque asset → `{ image: { url } }`.
  S'il n'y a **aucun** asset, retomber sur `visualUrl` (comportement actuel).
- Étendre `createPost` (`src/lib/buffer.ts`) pour accepter `imageUrls: string[]`
  (remplace `imageUrl: string` ; adapter l'appelant).
- Contrôles avant envoi (messages exacts, pattern d'erreur §2.4 du GUIDELINE) :
  - Instagram sans aucun média → **bloquant** (règle actuelle, conservée) ;
  - `format === "carrousel"` avec 0 ou 1 média → avertissement non bloquant renvoyé
    dans le résultat (« Un carrousel Buffer attend 2 à 10 images — envoyé avec {n} ») ;
  - plus de 10 médias → bloquant (« Instagram accepte 10 images maximum »).
  Aucune vérification de ratio/poids par analyse d'image — hors périmètre, et
  surtout : aucune transformation (règle d'Ana).
- **Cohabitation G11** : G11 §1 prévoit la colonne `buffer_post_id` et le stockage de
  l'id retourné. Si G11 n'est pas encore passé quand tu fais ce bloc, **fais-le ici**
  (migration `buffer_post_id` text nullable + stockage de `result.data.id`) et
  signale-le dans le commit pour que G11 §1 soit sauté. La purge §4 en dépend.

## 4. Purge automatique (le système ne s'encombre jamais)

Dans `/api/cron/quotidien` (piège §11.8) :

- Sélectionner les publications avec `buffer_post_id` non nul (= déjà envoyées à
  Buffer) et `planned_at` antérieur à **J-7**, qui ont encore des `publication_assets`.
- Pour chacune : `del(pathname)` de chaque asset (import `del` de `@vercel/blob`,
  try/catch par blob — un blob déjà absent ne doit pas stopper la purge), puis
  suppression des lignes `publication_assets`. Logger le total en console.
- Même logique dans `deletePublication` (action existante) : supprimer les blobs
  avant la suppression de la ligne (le cascade DB ne supprime que les lignes, pas
  les fichiers).
- Sans `BLOB_READ_WRITE_TOKEN` : la purge se saute silencieusement.

## 5. Envoi en lot : « Envoyer la semaine sur Buffer »

Sur `/planning`, à côté des actions existantes :

- Bouton « Envoyer la semaine sur Buffer » : cible les publications `status =
  "planifiee"`, sans `buffer_post_id`, avec `planned_at` dans les **7 prochains
  jours**. Bouton désactivé (avec compteur à 0) s'il n'y en a aucune.
- Clic → dialog de récapitulatif : liste `{date} · {marque} · {plateforme} · {titre}`,
  avec badge d'alerte sur celles sans média (« sans visuel » — elles partiront
  texte seul, sauf Instagram qui échouera : l'indiquer). Bouton de confirmation
  unique.
- Confirmation → action serveur qui appelle `envoyerSurBuffer` **séquentiellement**
  (pas de `Promise.all` : rate limit Buffer inconnu) et continue même si une
  publication échoue. Retour : rapport par ligne (✓ envoyée / ✗ message d'erreur),
  affiché dans le dialog.

## Critères d'acceptation

- [ ] Uploader 3 images sur une publication carrousel → le post Buffer créé contient
      les 3 images dans l'ordre choisi.
- [ ] Réordonner puis supprimer un asset fonctionne (ligne + blob supprimés).
- [ ] Une publication envoyée à Buffer et datée de plus de 7 jours voit ses blobs
      purgés au cron suivant ; une publication non envoyée garde les siens.
- [ ] Supprimer une publication supprime ses blobs.
- [ ] « Envoyer la semaine » avec 2 publications valides et 1 sans visuel Instagram :
      2 ✓, 1 ✗ avec message clair, rien n'est interrompu.
- [ ] Sans `BLOB_READ_WRITE_TOKEN` : le champ URL en repli fonctionne comme avant,
      aucune erreur au chargement.
- [ ] `npx tsc --noEmit` et `npm run build` passent.
